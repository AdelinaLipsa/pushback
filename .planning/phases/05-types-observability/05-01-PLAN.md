## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer shell → Supabase CLI → types file | Developer runs `supabase gen types` against live project; output written verbatim to repo |
| repo → TypeScript compiler | Generated types become part of the compile surface of every route handler and server component |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | I (Information disclosure) | types/database.types.ts checked into repo | accept | Supabase generated types contain only table shapes (column names, types) — these match the OSS migration SQL already in the repo. No secrets, no row data. Committing is required for typed-client compile correctness. |
| T-05-02 | T (Tampering) | gen:types script relies on `$SUPABASE_PROJECT_REF` env var | mitigate | Script uses shell env var, not a literal project ID. A wrong/attacker-controlled ref produces either a failing CLI call (auth required) or types that fail `types:check` against existing runtime code. Both are loud failures, not silent corruption. |
| T-05-03 | S (Spoofing) | Developer runs wrong supabase project (staging vs prod) | accept | Supabase CLI requires login + scoped access to the project ref. Wrong ref fails at the CLI layer. Solo-developer context: only one prod project exists. |
| T-05-04 | E (Elevation of privilege) | Database generic on admin client exposes typed RPC calls including `check_and_increment_*` | mitigate | Generic is type-only (compile-time). Runtime authority comes from service-role key handling already audited in Phase 2 (INFRA-01). Generic applies equally to server + admin clients — no new runtime power granted. |
</threat_model>
