# Phase 999.1: Client Reply Threading - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Freelancer pastes a client's reply to a sent pushback ‚Üí app classifies client stance (backing_down / doubling_down / escalating / unclear) using the original defense response as context ‚Üí generates a ready-to-send follow-up message tailored to the stance. Delivered via: new `reply_threads` Supabase table + migration, new API route `POST /api/responses/[id]/reply`, and an inline "Paste their reply" expansion in the existing `ResponseHistory` card on the history page. New `ReplyThreadCard` component shows the Lucide icon + label + follow-up text block with CopyButton.

</domain>

<decisions>
## Implementation Decisions

### Free-Tier Gating
- **D-01:** Reply analysis **consumes from the existing `check_and_increment_defense_responses` credit pool** ‚Äî same gate as `analyze-message`. Free users spend their 1 credit on whichever AI generation they make first. No separate gate, no Pro-only lock. Consistent mental model: every Anthropic call costs a credit.

### UI Entry Point
- **D-02:** Reply threading lives on the **history page only** (`/projects/[id]/history`). Not surfaced on the project detail / DefenseDashboard page.
- **D-03:** The "Paste their reply ‚Üí" button is **always visible** on every response card ‚Äî not conditional on `was_sent`. Freelancers may have sent via another channel without marking it.
- **D-04:** Clicking the button **expands a textarea inline** within the history card (collapsed by default). No modal, no navigation. Same expand/collapse pattern used elsewhere in the app.

### Risk Signal Display
- **D-05:** Signal is shown as a **Lucide icon + text label** (not a pill badge). Claude picks semantically appropriate icons per stance ‚Äî planner has discretion on exact icon names from Lucide React.
- **D-06:** Color coding by stance: lime (`--brand-lime`) for `backing_down`, amber/orange for `doubling_down` and `unclear`, red (`--urgency-high`) for `escalating`. Matches existing urgency color system in globals.css.
- **D-07:** Follow-up message is shown **below the signal label** in the same pre-formatted text block + `CopyButton` pattern used by `ResponseOutput`. No collapsed section ‚Äî full text visible immediately.

### Regeneration
- **D-08:** **One-shot in v1 ‚Äî no regenerate button.** Once a client reply is analyzed, the result is stored and the textarea is replaced by the analysis result. The unique constraint on `reply_threads` (`defense_response_id` UNIQUE) enforces this at the DB level.

### Architecture (locked by research ‚Äî carry forward)
- **D-09:** New `reply_threads` table (NOT additional columns on `defense_responses`). Schema per RESEARCH.md ‚Äî `id`, `defense_response_id` (FK + UNIQUE), `user_id`, `client_reply`, `risk_signal`, `signal_explanation`, `follow_up`, `created_at`. RLS: `auth.uid() = user_id`.
- **D-10:** `extractJson` stays **inline** in the new reply route (D-13 from STATE.md ‚Äî do not extract to a shared lib module).
- **D-11:** Compensating `decrement_defense_responses` RPC call on **every early return** after the gate ‚Äî Zod failure, 404, Anthropic error, schema validation failure, insert failure.
- **D-12:** Two-message Anthropic context: message 1 = `"ORIGINAL SITUATION:\n{situation}\n\nMESSAGE YOU SENT:\n{response}"`, message 2 = `"CLIENT REPLIED:\n{client_reply}"`. System prompt = `REPLY_ANALYSIS_SYSTEM_PROMPT` (new export from `lib/anthropic.ts`).
- **D-13:** `REPLY_ANALYSIS_SYSTEM_PROMPT` instructs JSON-only output with schema `{ risk_signal, signal_explanation, follow_up }`. Full draft in RESEARCH.md.
- **D-14:** On Postgres unique constraint violation (23505 / duplicate reply thread), return the **existing row** rather than an error ‚Äî upsert semantics or "conflict: do nothing, return existing" to handle double-click race condition.
- **D-15:** `await params` before destructuring route params (Next.js 16 breaking change ‚Äî established pattern in all existing routes).

### Claude's Discretion
- Exact Lucide icon names for each stance (TrendingDown, ChevronDown, AlertTriangle, HelpCircle ‚Äî or equivalents; planner picks from Lucide React)
- Exact label text for each stance (e.g., "Backing down", "Doubling down", "Escalating", "Unclear")
- Minor expand/collapse animation style for the reply textarea

</decisions>

<specifics>
## Specific Ideas

- The unique constraint `CREATE UNIQUE INDEX reply_threads_one_per_response ON reply_threads (defense_response_id)` enforces one-shot v1. On conflict (23505), return the existing row.
- Signal explanation is one sentence ‚Äî show it as a subdued text line between the icon+label and the follow-up block.
- The "Paste their reply ‚Üí" button text echoes the analyze-message flow's "Analyze message" language ‚Äî feel free to match that voice.
- History page Supabase join: `supabase.from('defense_responses').select('*, reply_threads(defense_response_id, risk_signal, signal_explanation, follow_up)').eq(...)` ‚Äî PostgREST nested select syntax, already verified as the project's Supabase pattern.

</specifics>

<deferred>
## Deferred Ideas

- **Multi-turn conversation (v2):** Allow multiple reply rounds per defense response (drop the unique constraint). Noted for a future phase if user demand warrants it.
- **Regeneration:** Allow generating a different follow-up for the same client reply (costs a credit). One-shot is v1; regeneration can be added in v2 with minimal code change (remove unique constraint, allow new inserts).
- **Pro-only gating for replies:** Could gate reply threading behind Pro plan with one extra plan check before the RPC gate. Deferred ‚Äî current decision is credit-pool consumption.

</deferred>
