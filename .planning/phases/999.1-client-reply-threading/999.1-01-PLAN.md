<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API → Postgres | Untrusted user input (`client_reply`) eventually persisted in `reply_threads` |
| auth → row-level security | RLS policy on `reply_threads` is the database-tier defense against cross-tenant reads |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-01 | Information Disclosure | reply_threads table (cross-tenant read) | mitigate | RLS policy `auth.uid() = user_id` enforced at table level via `for all using` clause; matches the existing `defense_responses` policy verbatim |
| T-999.1-02 | Tampering | reply_threads.user_id forgery | mitigate | `user_id` denormalized on the row + RLS policy means even a forged write attempt is rejected at the DB tier; route layer adds `.eq('user_id', user.id)` (defense-in-depth) implemented in plan 02 |
| T-999.1-03 | Tampering | Multiple reply threads per defense_response (race / double-click) | mitigate | UNIQUE index `reply_threads_one_per_response` enforces v1 invariant at DB tier; second concurrent INSERT receives 23505 error (handled in plan 02 as return-existing per D-14) |
| T-999.1-04 | Information Disclosure | Orphaned threads after parent defense_response deleted | mitigate | `ON DELETE CASCADE` on `defense_response_id` FK ensures threads are removed when parent is removed (matches `defense_responses → projects` pattern) |
| T-999.1-05 | Tampering | risk_signal column accepts any text (not enum-constrained) | accept | Zod validates `risk_signal` against the 4-value enum at the route layer; DB-level CHECK constraint deliberately omitted to match the existing `defense_responses.tool_type` precedent (text column, route validates) — Architecture Decision per RESEARCH.md A4 |
</threat_model>
