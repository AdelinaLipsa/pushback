**From `@react-pdf/renderer@^4.5.1` (installed in 15-02, verified in node_modules/@react-pdf/renderer/lib/react-pdf.d.cts):**
- `Document`, `Page`, `View`, `Text` Рђћ React-PDF host components (NOT DOM elements; `react-dom/server` cannot render them).
- `renderToBuffer(element): Promise<Buffer>` Рђћ server-side renderer. Implementation in `lib/react-pdf.js` line 320: `renderToStream(element).then(stream => collect into Buffer)`. Returns a Node Buffer ready for `new Response(buffer, ...)`.
- `renderToString` is deprecated in v4 (warning emitted on call) and only returns the Buffer toString'd Рђћ useless for snapshots.

**Snapshot strategy (decision: JSON tree walk, not renderToString).** Because React-PDF elements are not DOM elements, `react-dom/server.renderToString` throws/produces garbage. The maintainable approach is a tiny tree walker that converts the React element tree into a plain JSON shape `{ type: string, props: { ...serializable }, children: [...] }`. This pins composition + props + text content without depending on React-PDF's binary output (which embeds font-subset hashes and is non-deterministic across patch versions). This matches CONTEXT.md Claude's Discretion (e) Рђћ "JSON-of-component-tree" is the most maintainable option per D-19.

**Fixtures are pre-baked PackData objects, NOT calls into `assemblePackData`.** This decouples Plan 03's rendering tests from Plan 01's algorithm tests. `generatedAt` must be a fixed ISO string in every fixture Рђћ never `Date.now()` Рђћ so snapshots are stable across runs.
