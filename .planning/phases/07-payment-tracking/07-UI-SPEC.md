## Component Inventory

### Visual Hierarchy

Focal point per screen state — the single element the eye must land on first:

- Empty state focal point: "Save Payment Details" button (lime — the only colored element in the state; all other elements are neutral).
- Overdue state focal point: "Handle Late Payment" button (lime, `btnStyles.primary`, weight 700) — secondary focal point: the OVERDUE status line rendered in `--urgency-high` red, creating urgency contrast against the neutral card surface.
- Received state focal point: the "Received [date] · [currency] [amount]" status line in `--brand-green` — the only colored element in the state, communicating successful completion at a glance.

### New Component: PaymentSection

**File:** `components/project/PaymentSection.tsx`
**Type:** Client component (`'use client'`)

#### Empty State (no due date set — PAY-01 entry point)

Layout: Card with `--bg-surface` background, `1px solid var(--bg-border)` border, `borderRadius: var(--radius)` (0.625rem), `padding: 1.5rem`.

Elements:
1. Section heading: "Payment" — 1.25rem, weight 600, `--text-primary`
2. Subtext: "No payment details set." — 0.875rem, `--text-muted`
3. Inline form row with:
   - Date input: `<input type="date">` using `inputStyle` from `lib/ui.ts`; label "Due date" using `labelStyle`
   - Amount input: `<input type="number">` using `inputStyle`; label "Amount" using `labelStyle`
   - "Save Payment Details" button: `btnStyles.primary` (lime — this is the focal point CTA for the empty state, prompting the user to set payment details for the first time)
4. Loading state: "Save Payment Details" button disabled + text "Saving…" during PATCH request

#### Populated State (due date is set, payment not yet received — D-06)

Elements:
1. Section heading: "Payment" — same as above
2. Status line — one of three variants (see Color section for exact colors):
   - Not yet due: `"Due [formatted date] · [currency] [amount]"`
   - Overdue: `"OVERDUE · [N] days · [currency] [amount]"` in `--urgency-high`
3. When overdue: two action buttons on a row
   - "Handle Late Payment" — `btnStyles.primary` (lime, weight 700) — triggers scroll + pre-fill
   - "Mark as Received" — `btnStyles.outline`
4. Edit toggle: "Edit Payment" link/button using `btnStyles.ghost` reveals the inline form (same fields pre-filled, "Update Payment Details" + "Discard Changes" buttons)
5. Edit cancel: restores read view without saving

#### Received State (payment_received_at is non-null)

Elements:
1. Section heading: "Payment"
2. Status line: `"Received [formatted date] · [currency] [amount]"` in `--brand-green`
3. Edit toggle: "Edit Payment" in `btnStyles.ghost` to update payment details if needed; reveals inline form with "Update Payment Details" + "Discard Changes" buttons

#### Form Inputs

- Date input: `<input type="date">` — browser native date picker; styled with `inputStyle` from `lib/ui.ts`
- Amount input: `<input type="number" min="0" step="0.01">` — styled with `inputStyle`
- Currency: no currency selector in this phase — display as plain numeric with currency prefix derived from existing project data or defaulting to "EUR" (per D-07 example "EUR 3,000" in CONTEXT.md specifics)

> Source: D-05, D-06, D-08, D-09, CONTEXT.md Claude's Discretion (date input type = `<input type="date">`).

## Interaction Contracts

### "Handle Late Payment" Button

1. User clicks button in PaymentSection (overdue state).
2. Client-side: calculate `daysOverdue = Math.floor((Date.now() - new Date(payment_due_date).getTime()) / 86400000)`.
3. Select tier based on daysOverdue:
   - 0–7 → `payment_first`
   - 8–14 → `payment_second`
   - 15+ → `payment_final`
4. Build contextFields per tier (D-12):
   - `payment_first`: `{ invoice_amount: String(payment_amount), due_date: formatDueDate(payment_due_date) }` where `formatDueDate` returns "April 15" style (e.g. `new Date(payment_due_date).toLocaleDateString('en-US', { month: 'long', day: 'numeric' })`)
   - `payment_second` and `payment_final`: `{ invoice_amount: String(payment_amount), days_overdue: String(daysOverdue) }`
5. Call prop callback `onHandleLatePayment({ tool, contextFields })` — parent sets pre-fill state on DefenseDashboard.
6. Smooth scroll to DefenseDashboard: `document.getElementById('defense-dashboard')?.scrollIntoView({ behavior: 'smooth' })`.

> Source: D-10, D-11, D-12, D-13, CONTEXT.md specifics (smooth scroll, date format "April 15").

### "Mark as Received" Button

1. User clicks button in PaymentSection (overdue state only).
2. Button enters loading state ("Marking…", disabled).
3. PATCH `api/projects/[id]` with `{ payment_received_at: new Date().toISOString() }`.
4. On success: `router.refresh()` — overdue badge clears immediately across ProjectCard, ProjectHeader, and PaymentSection.
5. On error: display inline error message "Failed to mark as received. Please try again." in `--urgency-high` color below the button row. Button re-enables.

> Source: D-09, CONTEXT.md code_context (router.refresh() after mutations).

### Inline Edit Toggle (PaymentSection)

1. Read view shows "Edit Payment" ghost button.
2. Click "Edit Payment": replace status line + action buttons with pre-filled form (date + amount inputs) + "Update Payment Details" + "Discard Changes" buttons.
3. "Discard Changes": restore read view without calling API.
4. "Update Payment Details": PATCH `api/projects/[id]` with `{ payment_due_date, payment_amount }`. On success: `router.refresh()`. On error: inline error below the form.

> Source: D-06, CONTEXT.md code_context (Phase 4 inline edit toggle pattern).

### Scroll Anchor

The DefenseDashboard section must have `id="defense-dashboard"` set on its outermost wrapper element so the smooth scroll in step 6 of "Handle Late Payment" resolves correctly.
