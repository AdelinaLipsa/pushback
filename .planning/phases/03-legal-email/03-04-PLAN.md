<stripe_field_contract>
**CRITICAL — `current_period_end` is on `SubscriptionItem` in Stripe v22, NOT on `Subscription` root.**

Correct path (verified against `node_modules/stripe/cjs/resources/SubscriptionItems.d.ts` line 50):
```
const sub = await stripe.subscriptions.retrieve(session.subscription as string)
const periodEnd = sub.items.data[0]?.current_period_end  // number | undefined
```

**Incorrect path (silent undefined, causes fallback text to appear even when data exists):**
```
sub.current_period_end   // ← Does NOT exist on Subscription in v22
```

**Why this matters:** TypeScript will show `Property 'current_period_end' does not exist on type 'Subscription'` — so the type error is the guardrail. But some codemods / older examples put this field at the root. Research (PATTERNS.md note 1, RESEARCH.md Pitfall 1) flags this as the #1 silent bug risk in this phase.

**Date formatting:** `new Date(periodEnd * 1000).toLocaleDateString('en-US', { year: 'numeric', month: 'long', day: 'numeric' })` — yields strings like `"May 24, 2026"`.
</stripe_field_contract>

<billing_fallback_rule>
The upgrade email must send even if billing data is partial. Fallback logic (per D-08):

- If `session.amount_total` IS null OR `session.currency` IS null → `billing.amount = null`.
- Otherwise → `billing.amount = '${(session.amount_total / 100).toFixed(2)} ${session.currency.toUpperCase()}'` (e.g. `"9.00 USD"`).
- If `stripe.subscriptions.retrieve()` throws OR `sub.items.data[0]?.current_period_end` is undefined → `billing.nextBillingDate = null`.
- Otherwise → `billing.nextBillingDate = new Date(periodEnd * 1000).toLocaleDateString('en-US', { year: 'numeric', month: 'long', day: 'numeric' })`.

The `lib/email.ts` `upgradeHtml()` template (Plan 03-02) handles `null` values by rendering the Stripe-dashboard fallback paragraph. The webhook does NOT need to short-circuit the email send when billing is missing.
</billing_fallback_rule>

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-17 | Tampering | Forged webhook → unauthorized upgrade email | mitigate | Existing `stripe.webhooks.constructEvent(body, sig, STRIPE_WEBHOOK_SECRET)` at line 15 rejects unsigned / invalid events with 401. | T-03-19 | Information Disclosure | Wrong user's email queried for upgrade | mitigate | `userId` comes from `session.metadata.user_id` which was set by `createCheckoutSession()` in `lib/stripe.ts` at checkout creation time — server-authoritative. `.eq('id', userId).single()` is the same filter used for the existing DB update at line 35, so a mismatch is impossible without a pre-existing bug. |
| T-03-20 | Information Disclosure | Stripe API error details leaked to clients | mitigate | `stripe.subscriptions.retrieve()` is wrapped in try/catch; only `console.error` logs the error. The webhook response body is always `{ received: true }` or `{ error: 'Database update failed' }` (existing) — never Stripe's error payload. |
| T-03-21 | Denial of Service | Stripe API outage during retrieve → webhook timeout | mitigate | The try/catch around `stripe.subscriptions.retrieve()` absorbs the failure; `billingDetails.nextBillingDate` stays null, email still sends with fallback copy. No upstream timeout propagates to Stripe's webhook delivery. |
| T-03-22 | Denial of Service | Resend email failure → webhook 5xx → Stripe retry storm | mitigate | Email is fire-and-forget via `.catch()`. Resend failure is logged but never throws to the outer scope. Webhook always returns 200. No retry storm possible. |

| T-03-24 | Tampering | `session.subscription` is a hydrated Subscription object (not string) — `as string` cast silently wrong | mitigate | The existing code at line 33 already uses `session.subscription as string` for the DB update, and Stripe's webhook payload for `checkout.session.completed` in subscription mode always delivers the ID as a string (Stripe's `expand` feature is opt-in).
