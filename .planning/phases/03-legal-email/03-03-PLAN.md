<new_user_detection>
A user is "new" iff `Date.now() - new Date(data.user.created_at).getTime() < 60_000` (60 seconds).

**Why 60s:** The auth callback fires:
1. On FIRST completion of email-confirmation signup — `created_at` is seconds ago → isNewUser TRUE.
2. On FIRST completion of Google OAuth signup — `created_at` is seconds ago → isNewUser TRUE.
3. On every Google OAuth sign-IN (returning user) — `created_at` is minutes/days ago → isNewUser FALSE.
4. On every password-reset flow that re-runs the callback (rare) — `created_at` is historical → isNewUser FALSE.

60s is a safe window: longer than network + Supabase processing (usually <5s), shorter than any returning-login scenario. Sourced from CONTEXT.md Specifics line: `const isNewUser = Date.now() - new Date(user.created_at).getTime() < 60_000`.
</new_user_detection>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Anonymous browser → auth callback GET | The callback receives a Supabase one-time code over the query string; boundary established pre-phase. |
| Auth callback → Resend API | NEW in this plan — outbound email send crosses this boundary via `lib/email.ts`. |
| Auth callback → Supabase Auth | Existing boundary; `exchangeCodeForSession` exchanges the code for a session. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-12 | Information Disclosure | Email enumeration via response timing | mitigate | Email is fire-and-forget (never awaited) → Resend latency does not affect callback response time → no timing side-channel distinguishes new signups from returning logins via network observation. |
| T-03-13 | Tampering | Forged `created_at` to trigger welcome emails on returning users | mitigate | `data.user.created_at` comes from Supabase's server-authoritative session payload, not from the URL or a client-controllable input. Attacker cannot forge it via query-string manipulation. |
| T-03-14 | Denial of Service | Resend rate-limit exhaustion via rapid signups | accept | Low risk at current scale; Supabase's own signup rate limits cap upstream signup rate. Per-user rate limiting is OPS-02 (v2, not v1). Accepted. |
| T-03-15 | Information Disclosure | Welcome email sent to wrong address due to stale session | mitigate | `data.user.email` comes from the freshly-exchanged session — not cached. If `data.user.email` is `undefined` (edge case for phone-only auth), the `if (isNewUser && data.user.email)` guard skips the send. |
| T-03-16 | Repudiation | Silent email failure — user never receives welcome | accept | `console.error('Welcome email failed:', err)` logs the failure. Full observability (Sentry) comes in Phase 5 (OBS-01). Accepted for v1; welcome email is non-critical (user is already on the dashboard). |
