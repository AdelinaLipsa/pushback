<!-- Existing font wiring pattern in app/layout.tsx -->

Current `app/layout.tsx` already proves the `next/font/google` pattern that Plan 16-01 extends:

```tsx
import { Geist, Geist_Mono } from 'next/font/google'

const geist = Geist({
  subsets: ['latin'],
  variable: '--font-inter',
})

const geistMono = Geist_Mono({
  subsets: ['latin'],
  variable: '--font-mono',
})

// ...
<html lang="en" className={`${geist.variable} ${geistMono.variable} h-full antialiased`}>
```

Plan 16-01 adds `Instrument_Serif` to the same import, declares an `instrumentSerif` constant with `variable: '--font-serif'`, and appends `${instrumentSerif.variable}` to the `<html>` className template. The `Instrument_Serif` Google font is **weight-locked at 400** (it has no other weights on Google Fonts), so the loader must pass `weight: '400'`. `next/font/google` defaults to `display: 'swap'` so no extra option is required.

<!-- HeroArtifactSVG consumer signature -->

The downstream consumer (PushbackHero, Plan 16-02) will render the artifact like:

```tsx
import HeroArtifactSVG from '@/components/hero/HeroArtifactSVG'

<HeroArtifactSVG className="hero-artifact" />
```

So the component must accept an optional `className` prop and forward it to the root `<svg>` element. No other props.

<!-- CSS variable consumption convention -->

Per project memory `feedback_lime_color.md` and existing codebase conventions, lime is the canonical accent: `#84cc16` literal OR `var(--brand-lime)`. Inside the SVG, use the hex literal directly so the SVG renders correctly when copied/exported in isolation (CSS variables resolve only inside the document tree).

Use a small array literal in JSX (`[280, 420, 380, 456, 312, ...].map((w, i) => <rect ... />)`) to keep the source compact.
